# LangChain

LLM 기반 애플리케이션 개발을 위한 프레임워크

## 정의

> 모델별로 다른 호출 방식과 기능을 공통된 인터페이스로 다룰 수 있도록 만든 프레임워크

즉, Claude, GPT, Gemini 등 서로 다른 모델을 비슷한 방식으로 연결하고 사용할 수 있도록 돕는 도구

## 등장 배경

LLM마다 SDK, 호출 방식, 메시지 포맷, 응답 처리 방식이 서로 다름

이로 인해 모델을 교체하거나 여러 모델을 함께 사용할 때 코드 구조가 복잡해지는 문제 발생

LangChain은 이러한 차이를 줄이고, 모델과 무관하게 기능 중심으로 개발할 수 있도록 공통 인터페이스 제공

핵심 목적은 다음과 같음

```text
모델별 차이 추상화
→ 개발 방식 통일
→ 모델 교체 비용 감소
→ 애플리케이션 구성 요소 재사용 용이
```

## 구성

### 1. Core

모델 종류와 무관하게 공통 방식으로 개발할 수 있도록 하는 핵심 영역

주요 역할

- 메시지 구조 정의
- 프롬프트 처리
- 체인 구성
- 실행 흐름 관리
- 공통 인터페이스 제공

### 2. Integration

각 모델 및 외부 도구와 실제로 연결되는 영역

예시

- OpenAI
- Anthropic
- Google
- Vector DB
- 외부 툴
- Retriever

즉, Core가 공통 구조를 담당하고, Integration이 실제 서비스나 모델 연결을 담당하는 구조

## LangChain 사용 전후 비교

### LangChain 없이 개발하는 경우

모델마다 서로 다른 라이브러리와 호출 방식을 직접 다뤄야 하는 구조

```text
Claude 사용
→ anthropic 방식으로 구현

GPT 사용
→ openai 방식으로 구현

모델 변경 시
→ 호출 코드, 메시지 형식, 응답 처리 로직 수정 필요
```

### LangChain을 사용하는 경우

모델 객체만 바꾸고 나머지 흐름은 유사하게 유지 가능한 구조

```python
llm = ChatAnthropic()
llm = ChatOpenAI()
llm = ChatGoogleGenerativeAI()

response = llm.invoke("안녕")
```

즉, 모델 교체 시 변경 범위를 줄일 수 있는 구조

## LangChain이 제공하는 주요 기능

### 1. 모델 연결

여러 LLM을 공통된 방식으로 호출 가능

예시

- Claude
- GPT
- Gemini

즉, 모델별 SDK 차이를 직접 처리하지 않고 통일된 방식으로 사용 가능한 구조

### 2. 프롬프트 관리

프롬프트를 템플릿 형태로 관리 가능한 구조

장점

- 재사용 용이
- 변수 주입 가능
- 프롬프트 구조 일관성 유지

예시 개념

```text
질문: {question}
문맥: {context}
형식: {format}
```

즉, 하드코딩보다 관리성이 높은 구조

### 3. 체인(Chain)

여러 작업을 순서대로 연결하는 기능

예시 흐름

```text
문서 읽기
→ 요약
→ 번역
→ 최종 응답 생성
```

즉, 하나의 LLM 호출이 아니라 여러 단계를 연결한 파이프라인 구성 가능

### 4. 메모리

대화 이력이나 이전 문맥을 관리하는 기능

예시

- 이전 질문 기억
- 사용자 이름 유지
- 직전 대화 맥락 반영

다만 최근에는 단순 메모리보다 명시적인 메시지 관리 방식이나 상태 관리 방식이 더 자주 사용되기도 함

### 5. RAG 연결

Retriever, Vector DB, LLM을 연결해 검색 기반 응답 시스템 구성 가능

예시 흐름

```text
질문 입력
→ 관련 문서 검색
→ 검색 결과를 프롬프트에 삽입
→ LLM 응답 생성
```

즉, RAG 프로토타입 구현에 유리한 구조

## 장점과 단점

### 장점

```text
빠른 프로토타이핑
모델 교체 용이
프롬프트 체인 구성 편리
RAG 연결 편리
구성 요소 재사용 가능
```

### 단점

```text
추상화 수준이 높음
내부 동작 파악 어려움
버전 변화 잦음
단순한 프로젝트에는 오히려 과한 구조
디버깅 복잡성 증가 가능
```

## 실무 관점 정리

프로젝트 규모와 복잡도에 따라 선택이 달라짐

```text
단순한 호출 중심 프로젝트
→ 모델 API 직접 호출

여러 단계의 처리 흐름 필요
RAG, 체인, 툴 연결 필요
모델 교체 가능성 존재
→ LangChain 활용 고려
```

즉, LangChain은 모든 프로젝트의 필수 도구라기보다 복잡한 LLM 애플리케이션을 빠르게 구성하기 위한 프레임워크로 보는 편이 적절

## System Message vs User Message

## 기본 개념

LLM과 대화할 때 전달되는 메시지는 역할에 따라 구분 가능

가장 대표적인 구분이 System message와 User message

## 차이

```text
System message
→ AI의 역할, 규칙, 응답 스타일 설정

User message
→ 사용자가 실제로 입력하는 질문이나 요청
```

## 비교

| 구분 | System Message | User Message |
| --- | --- | --- |
| 작성 주체 | 개발자 | 사용자 |
| 입력 시점 | 대화 시작 전 또는 상위 제어 단계 | 실제 대화 시점 |
| 역할 | 모델의 행동 규칙 정의 | 실제 요청 전달 |
| 사용자 노출 여부 | 일반적으로 숨김 | 사용자에게 보임 |

## System Message의 의미

System message는 모델이 어떤 역할로 응답해야 하는지, 어떤 규칙을 우선해야 하는지 정하는 상위 지침

예시

```text
너는 부동산 전문가 역할 수행
너는 요리 전문가 역할 수행
너는 개화율 분석가 역할 수행
```

같은 기반 모델이라도 System message에 따라 응답의 방향성과 스타일이 달라질 수 있음

즉, System message는 모델을 서비스 목적에 맞게 조정하는 핵심 수단

## 실제 활용 관점

System message로 주로 설정하는 항목

- 역할
- 말투
- 출력 형식
- 금지 사항
- 우선순위
- 답변 범위
- 안전 규칙

예시

```text
항상 표 형태로 답변
모르면 추측하지 말 것
한국어로만 답변
3문장 이내로 요약
```

즉, 서비스 수준의 일관성을 만드는 데 중요한 요소

## Temperature

## 정의

모델이 다음 토큰을 선택할 때의 무작위성 수준을 조절하는 값

즉, 응답의 일관성과 다양성에 영향을 주는 파라미터

## 동작 개념

Temperature가 낮을수록 확률이 높은 토큰을 더 우선적으로 선택

Temperature가 높을수록 낮은 확률의 토큰도 선택될 가능성 증가

즉, 값이 낮으면 보수적 응답, 높으면 다양성이 큰 응답 경향

## 예시

### Temperature = 0

가장 가능성이 높은 토큰 중심 선택

특징

- 응답 일관성 높음
- 반복 실행 시 유사한 결과
- 보수적 출력

적합한 예시

- 코드 생성
- 수학 계산
- 형식화된 답변
- 일관성 중요한 시스템 응답

### Temperature = 0.7

적당한 다양성과 자연스러움

특징

- 너무 경직되지 않음
- 대화형 응답에 적합
- 일반적인 챗봇 서비스에서 자주 사용

### Temperature = 1 이상

무작위성 증가

특징

- 더 다양한 표현 가능
- 창의적 결과 증가 가능
- 비논리적이거나 불안정한 응답 가능성도 증가

적합한 예시

- 창작
- 아이디어 발산
- 문장 스타일 다양화

## 서비스별 활용 예시

```text
분석 서비스
→ 낮은 Temperature
→ 일관된 결과 우선

시나리오 생성
→ 높은 Temperature
→ 표현 다양성 우선

코드 생성
→ Temperature 0 또는 낮은 값
→ 재현 가능성과 정확성 우선
```

## Temperature 외에 함께 보는 값

### 1. Top-p

누적 확률 기준으로 후보 토큰 범위를 제한하는 방식

즉, 확률이 높은 토큰들 중 상위 확률 질량만 남기고 그 안에서 선택하는 구조

### 2. Top-k

확률이 높은 상위 k개 토큰만 후보로 두는 방식

즉, 후보 수 자체를 제한하는 방식

### 3. Seed

무작위 선택 과정의 시작점을 고정하는 값

같은 모델, 같은 입력, 같은 설정, 같은 Seed라면 결과 재현성이 높아질 수 있음

다만 실제 서비스에서는 모델 서버 환경이나 구현 방식에 따라 완전 동일성이 항상 보장되지는 않음

## 정리

동일한 프롬프트라도 샘플링 설정에 따라 결과가 달라질 수 있음

```text
Temperature 낮음
→ 일관성 증가
→ 예측 가능한 응답

Temperature 높음
→ 다양성 증가
→ 창의적이지만 불안정할 수 있는 응답
```

즉, Temperature는 모델의 성격을 바꾸는 값이라기보다 토큰 선택 과정의 확률적 특성을 조정하는 파라미터로 이해하는 것이 적절

## 전체 요약

```text
LangChain
→ LLM 애플리케이션 개발용 프레임워크
→ 모델 호출, 프롬프트, 체인, RAG 연결 지원

System message
→ 모델의 역할과 규칙 정의

User message
→ 사용자의 실제 요청 전달

Temperature
→ 응답 생성 시 무작위성 조절

Top-p / Top-k
→ 토큰 후보 범위 조절

Seed
→ 결과 재현성 보조
```
