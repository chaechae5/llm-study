## 1. AI Agent 개요

### 패러다임의 전환: 예측 → 문제 해결

기존 AI(Predictive AI)는 질문 답변, 텍스트 번역, 이미지 생성 등 **수동적이고 단발성 작업**에 특화된 구조. 매번 사람의 지시가 있어야만 다음 작업을 수행할 수 있음.

반면 **Agentic AI**는 목표 달성을 위해 스스로 계획하고 행동하는 자율적 애플리케이션.

| 구분 | Predictive AI | Agentic AI |
|------|--------------|------------|
| 동작 방식 | 입력 → 출력 (단방향) | 목표 → 계획 → 행동 → 관찰 (순환) |
| 자율성 | 낮음 (매번 사람이 지시) | 높음 (스스로 다음 단계 판단) |
| 적합 작업 | 단발성, 수동적 작업 | 복잡한 목표 달성, 다단계 문제 해결 |

> 💡 **핵심 차이**: Predictive AI는 "도구", Agentic AI는 "자율적 행위자"

## 2. AI Agent 구성 요소

AI Agent는 세 가지 핵심 축으로 구성됨.

### 🧠 Model (두뇌)
- 정보를 처리하고 대안을 평가하여 **결정을 내리는 추론 엔진(LM)**
- 언제, 무엇을, 왜 할지 판단하는 중심 역할
- GPT-4, Claude, Gemini 등의 LLM이 이 역할을 담당

### 🛠️ Tools (손과 발)
- Agent의 추론을 외부 세계와 연결하는 인터페이스
- 텍스트 생성을 넘어 **실제 행동**을 가능하게 함
- 주요 유형:

| 도구 유형 | 예시 |
|----------|------|
| 데이터 조회 | DB 검색, 파일 읽기, 웹 검색 |
| 외부 시스템 연동 | REST API 호출, 이메일 발송, 슬랙 메시지 |
| 코드 실행 | Python 실행, SQL 쿼리 |
| 상태 저장 | 벡터 DB 저장, 캐시 관리 |

### 🔗 Orchestration (신경계)
- 에이전트의 **운영 루프를 관리**하는 핵심 프로세스
- 계획 수립, 메모리 관리, 추론 전략 실행을 담당
- **언제 사고하고 언제 도구를 사용할지** 결정

```
[사용자 목표 수신]
       ↓
  Orchestration
  ┌────────────────────────────────┐
  │  1. 목표 분해 (Planning)        │
  │  2. 메모리 참조 (Memory)         │
  │  3. 도구 선택 및 호출 (Tool Use) │
  │  4. 결과 평가 및 재계획           │
  └────────────────────────────────┘
       ↓
  [최종 응답 생성]
```

## 3. Think, Act, Observe 사이클

Agent는 목표가 달성될 때까지 다음 루프를 반복함.

```
User Goal
    ↓
[Think Phase] → 목표 분석 및 다단계 계획 수립
    ↓
[Execution Loop]
    Act     → 도구 호출 (API, 함수 등)
    Observe → 결과 관찰 및 다음 행동 결정
    ↓ (목표 달성 시까지 반복)
[Final Output]
```

### ReAct 패턴

Think-Act-Observe 사이클의 구현 패턴 중 하나가 **ReAct(Reasoning + Acting)**. LLM이 추론 과정(Thought)을 명시적으로 출력하고, 그에 따른 행동(Action)과 관찰(Observation)을 반복하는 방식.

```
Thought: 사용자가 주문 상태를 묻고 있음. 먼저 주문 정보를 조회해야 함.
Action: find_order("12345")
Observation: {"order_id": "12345", "tracking_no": "ZYX987", "status": "in_transit"}

Thought: 추적 번호를 얻었음. 배송 API를 통해 상세 상태를 조회해야 함.
Action: get_shipping_status("ZYX987")
Observation: {"status": "배송 중", "location": "수원 물류센터", "eta": "내일 오후"}

Thought: 모든 정보를 수집. 사용자에게 응답 가능.
Final Answer: 주문 #12345는 현재 수원 물류센터에서 배송 중이며, 내일 오후 도착 예정.
```

### 실제 예시: 주문 상태 조회

**사용자 목표:** "나의 주문 #12345의 상태는?"

**Think Phase (계획 수립):**
1. 내부 DB에서 주문 상세 정보 검색
2. 추적 번호를 추출하여 외부 배송 API 조회
3. 수집된 정보를 취합하여 응답 생성

**Execution Loop:**

| 사이클 | Act | Observe |
|--------|-----|---------|
| Cycle 1 | `find_order("12345")` 호출 | 추적 번호 'ZYX987'을 포함한 전체 주문 정보 획득 |
| Cycle 2 | `get_shipping_status("ZYX987")` 호출 | '배송 중' 상태 정보 획득 |

**Final Output:** "고객님의 주문 #12345는 현재 '배송 중'."

## 4. AI Agent 진화 단계

Agent의 성숙도를 4단계 레벨로 분류.

### Level 0: The Core Reasoning System
- 사전 학습된 지식만으로 응답하는 **외부와 고립된 언어 모델**
- 실시간 정보에는 사실상 '눈이 먼' 상태
- 예) "2023년 기준 최신 뉴스를 알려줘" → 학습 컷오프 이후 정보 없음

### Level 1: The Connected Problem-Solver
- 실시간 정보를 얻기 위해 **외부 도구(예: 검색 API)와 연결**
- 단순 도구 사용 수준의 자율성
- 예) 웹 검색 도구를 통해 최신 정보 조회 가능

### Level 2: The Strategic Problem-Solver
- 단순 작업을 넘어 **복잡하고 다단계 목표를 스스로 계획**
- 핵심 역량: 각 단계에 필요한 정보를 관리하는 **컨텍스트 엔지니어링**
- 예) "경쟁사 분석 보고서를 작성해줘" → 검색 → 데이터 수집 → 분석 → 보고서 생성을 자율 수행

### Level 3: The Collaborative Multi-Agent System
- 인간 조직처럼 **전문가 팀** 형태로 에이전트들이 서로 협업
- 각 에이전트가 역할을 나눠 작업을 위임하고 협력
- 예) PM Agent가 하위 Marketing, Dev, Research Agent에게 역할 분배

## 5. 멀티 에이전트 시스템

### 핵심 철학: 슈퍼 에이전트 → 전문가 팀

하나의 전지전능한 '슈퍼 에이전트'를 만드는 것보다 **인간 조직처럼 전문화된 에이전트들이 협력**하는 구조가 더 효과적.

### 멀티 에이전트 토폴로지

| 구조 | 설명 | 적합 상황 |
|------|------|----------|
| **Supervisor** | 중앙 Agent가 하위 Agent에게 지시 | 명확한 계층 구조, 총괄이 필요한 작업 |
| **Peer-to-Peer** | Agent들이 수평적으로 직접 통신 | 유연한 협업, 특정 전문 영역 간 조율 |
| **Pipeline** | Agent A → Agent B → Agent C 순서로 결과 전달 | 순차적 처리가 필요한 작업 |

### 멀티 에이전트 구성 예시: 헤드폰 신제품 출시

```
Project Manager Agent (총괄)
    ├── Marketing Agent
    │   └── 제품 사양서 기반 보도자료 초안 3가지 버전 작성
    ├── WebDev Agent
    │   └── 디자인 목업 기반 신제품 페이지 HTML 생성
    └── Market Research Agent
        └── 경쟁사 가격 분석 리포트 생성
```

각 Agent는 독립적으로 작업을 수행하고, PM Agent가 결과를 종합하여 최종 출시 패키지를 완성.

### 멀티 에이전트의 장점

- 각 에이전트가 **더 단순하고 명확한 책임**을 가짐 → 프롬프트/로직이 집중됨
- **독립적 테스트 및 유지보수** 가능
- 각 에이전트의 전문성 극대화
- 장애 발생 시 **격리(isolation)** 용이

### 주의할 점 ⚠️

- **오케스트레이션 복잡도** 증가: 에이전트 간 통신, 오류 처리, 재시도 로직 필요
- **컨텍스트 공유 방식** 설계 필요: 어떤 정보를 어떤 Agent와 공유할지 명확히 정의
- **무한 루프** 방지: 에이전트 간 순환 호출이 발생하지 않도록 종료 조건 설계 필수

## 6. Agent 운영 방안 (AgentOps)

프로덕션 환경에서 Agent를 안정적으로 운영하기 위한 4가지 핵심 방안.

### 1️⃣ Measurement (측정)

**문제:** 정답이 하나로 정해져 있지 않을 때, 품질을 어떻게 평가할까?

**해결책:** `LM-as-Judge` (언어모델 판정기) 활용

- 사전 정의된 평가 기준(Rubric)과 골든 데이터셋 준비
- 에이전트 출력물을 별도의 LM이 채점

```
Agent Output  ──┐
                ├──→ LM as Judge ──→ Quality Score
Golden Dataset ─┘
& Rubric
```

**평가 지표 예시:**

| 지표 | 설명 |
|------|------|
| Faithfulness | 생성된 답변이 사실에 근거한 정도 |
| Relevance | 질문과 답변의 관련성 |
| Completeness | 요청된 작업을 얼마나 완전히 수행했는가 |
| Tool Precision | 불필요한 도구 호출 없이 정확하게 도구를 사용했는가 |

### 2️⃣ Debugging (디버깅)

**문제:** 보이지 않는 문제를 어떻게 고칠 수 있을까?

**해결책:** `OpenTelemetry 트레이스` 활용

- 에이전트의 전체 실행 경로를 단계별로 상세 기록
- 에이전트의 **"사고 과정(thought process)"** 가시화

```
Step 1 → Step 2 → External Tool Call → Step 3 → Final Output
(각 단계별 Duration, Input/Output, Context data 기록)
```

> 💡 LangSmith, Langfuse, Arize Phoenix 등의 도구가 OpenTelemetry 기반 에이전트 트레이싱을 지원

### 3️⃣ Deployment (배포)

**문제:** 변경 사항을 어떻게 안전하게 배포할 수 있을까?

**해결책:** 메트릭 기반 + A/B 배포

- 새로운 버전을 전체 평가 데이터셋에 실행하여 품질 점수 비교
- 점진적 출시를 위해 **A/B 배포** 활용

```
Version A (Current) ─┐
                      ├──→ Evaluation Dataset ──→ Quality Comparison ──→ A/B Deployment
Version B (New)    ──┘
```

### 4️⃣ Improvement (개선)

**문제:** 루프를 어떻게 완성할 수 있을까?

**해결책:** 인간 피드백 기반 지속 개선

- 사용자가 보고한 문제를 수집 → **평가 데이터셋에 새 테스트 케이스로 추가**
- 회귀(Regression) 방지를 통한 품질 유지
- 이 과정을 반복하여 에이전트가 점진적으로 개선됨

## 7. Agent 보안 설계

> ⚠️ **보안은 사후에 덧붙일 요소가 아니라, 처음 설계부터 포함해야 함**

### Agent 정체성: 새로운 유형의 주체

에이전트는 **사용자도, 서비스도 아닌** 자율적 행위자. 기존 보안 모델로는 적절히 다루기 어려움.

- **SPIFFE 기반** 등 자체적으로 검증 가능한 정체성 필요
- 에이전트를 호출한 사용자, 개발자와 구별되는 독립적 정체성 부여
- **최소 권한 원칙(Least Privilege)**: 필요한 권한만 부여하여 침해 시 피해 범위 제한

### 주요 보안 위협

| 위협 | 설명 | 대응 |
|------|------|------|
| **Prompt Injection** | 악의적 입력이 Agent의 행동을 조작 | 입력 검증, 가드레일 적용 |
| **Tool Abuse** | 권한 밖의 도구를 호출하거나 과도한 권한 행사 | 도구별 권한 제한, 감사 로그 |
| **Data Exfiltration** | Agent가 민감 데이터를 외부로 유출 | 출력 필터링, 네트워크 정책 |
| **Runaway Loop** | 종료 조건 없이 무한 실행 | 최대 반복 횟수, 타임아웃 설정 |

### 다중 방어 (Defense in Depth)

AI 모델의 판단에만 의존하지 말고, **두 개 이상의 보호 계층**을 결합해야 함.

| 레이어 | 유형 | 설명 | 예시 |
|--------|------|------|------|
| Layer 1 | 결정적 가드레일 | 모델의 추론 영역 밖에서 동작하는 하드코딩된 비즈니스 규칙 | 정책 엔진이 100만 원 이상의 구매를 자동 차단 |
| Layer 2 | 추론 기반 방어 | 작고 특화된 '가드 모델'이 실행 전에 계획을 검사 | 위험하거나 정책 위반 단계를 사전에 표시 |

## 8. Agent 거버넌스

### 무분별한 Agent 확산 방지: 중앙 컨트롤 플레인

Agent와 Tool이 조직 전반에 무질서하게 확산되는 것을 방지하려면 **중앙 Gateway**가 필수.

```
Users  ──┐
          ├──→ [Gateway] ──→ Tools
Agents ──┘
```

### 중앙집중형 거버넌스 (Centralized Governance)

- **메커니즘:** 중앙 레지스트리 — 에이전트와 도구를 위한 엔터프라이즈 앱 스토어
- **기능:**
  - 개발자가 기존 자산을 쉽게 발견하고 재사용 가능
  - 관리자는 버전 관리와 세밀한 접근 정책 설정을 위한 전체 인벤토리 확보

### 런타임 정책 집행 (Runtime Policy Enforcement)

- **메커니즘:** 단일 아키텍처 초크포인트(Chokepoint)
- **기능:**
  - 인증("이 행위자는 누구인가?") + 인가("권한이 있는가?") 처리
  - 공통 로그, 메트릭, 트레이스를 생성하여 **단일 관점(Single Pane of Glass)** 제공

### 거버넌스 체크리스트

- [ ] 조직 내 모든 Agent와 Tool이 중앙 레지스트리에 등록되어 있는가?
- [ ] 각 Agent에 최소 권한 원칙이 적용되어 있는가?
- [ ] 모든 Tool 호출이 감사 로그에 기록되는가?
- [ ] Agent 별 비용(토큰, API 호출) 모니터링이 가능한가?
- [ ] 장애 발생 시 특정 Agent를 즉시 비활성화할 수 있는가?

## 9. 자율 학습하는 미래의 Agent

적응력이 없으면 Agent 성능은 시간이 지남에 따라 저하됨. Agent가 **자율적으로 학습하고 진화**하도록 설계해야 함.

### 학습 루프 구조

```
1. Agent System     → 작업 수행
        ↓
2. Critiquing Agent → 출력 결과 리뷰 / 필요 시 Human Expert에게 검토 요청
        ↓
3. Human Domain Expert → 수정된 피드백 제공
        ↓
4. Learning Agent   → 상호작용 관찰 → 피드백을 가이드라인으로 일반화
        ↓ (루프 반복)
```

이 루프를 통해 Agent는 **비즈니스 요건에 적합하도록 자율적으로 적응**.

### 자율 학습의 핵심 요건

- **피드백 수집 체계**: 사용자 평가, 전문가 리뷰, 자동 평가 지표 조합
- **가이드라인 자동 갱신**: 피드백을 구조화된 규칙으로 변환하는 Learning Agent 역할
- **인간 감독(Human-in-the-Loop)**: 완전 자율화 전까지는 중요 결정에 사람의 승인 필요
- **드리프트 감지**: 시간이 지나면서 성능이 저하되는 것을 조기에 탐지하는 모니터링

## 10. 핵심 요약

| 항목 | 내용 |
|------|------|
| **정의** | 목표 달성을 위해 스스로 계획·행동·관찰하는 자율적 AI 시스템 |

| **보안 원칙** | 최소 권한 + 다중 방어(가드레일 + 가드 모델) |
| **거버넌스** | 중앙 레지스트리 + 단일 Gateway + 감사 로그 |
